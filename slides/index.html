01

Working with the LLM API

The raw material: talking to a model over HTTP
01

The chat API is a stateless conversation

You send a list of messages with roles — system (sets behavior), user (the ask), assistant (past replies) — and get the next reply. The model remembers nothing between calls; you resend the history every time.

Analogy

Texting a brilliant consultant with amnesia. Each message, you paste the whole prior chat so they're caught up.

Aha

"Memory" is an illusion you build by re-sending context. The model is a pure function: messages in → reply out.

01

Tokens are the currency

Models read text in tokens — chunks roughly ¾ of a word. Pricing, rate limits, and context windows are all counted in tokens, not words or characters.

Analogy

Minutes on a phone plan. "OK" is 1 token; "antidisestablishmentarianism" is ~6. You're billed per token, not per word.

Aha

The same sentence in Japanese can cost ~2× the tokens of English — so language choice affects cost. (OpenAI tiktoken cookbook)

01

System prompt + temperature: the two dials

The system prompt defines who the model is (priority instructions). Temperature controls randomness: 0 = focused & near-deterministic, high = creative & surprising.

Analogy

System prompt = an actor's character brief. Temperature = how loaded the dice are — 0 always rolls a 6, 1.5 is a fair die.

Aha

Same question + different system prompt = a different personality. You reshape behavior without touching the user's words.

01

Streaming: same speed, better feel

Instead of waiting for the whole answer, streaming sends tokens as they're generated (server-sent events). Generation isn't faster — but it feels instant.

Analogy

Netflix vs. waiting for a full movie download. You start watching while the rest buffers.

Aha

Streaming cuts perceived latency, not real latency. The UX win is psychological — and huge.

01

Tool calling: the model never runs your code

You describe functions (name, purpose, parameters). When useful, the model returns structured JSON requesting a call. Your code runs it and passes the result back, which the model uses to finish.

Analogy

A boss shouting "get me Tokyo's weather!" through a closed door. You fetch it, slide a note back — they never leave the office.

Aha

The model emits a request, not an execution. You keep full control to validate, log, or refuse. (OpenAI function-calling docs)

01

Structured output & one client for all providers

Structured outputs force replies to match a schema exactly — no missing keys, safe to feed into typed code. A provider-agnostic interface hides differences between OpenAI / Anthropic / others behind one call.

Analogy

Schema = a form with a pen that only writes inside the boxes. Provider interface = a universal remote for any TV brand.

Aha

Abstraction isn't just tidy — it's freedom from lock-in. Swap models with a config change when one is slow or pricey.

01

The idea, in code

calling the chat API

response = client.chat.completions.create(
    model="gpt-4o-mini",        # which brain to use (project 01's default)
    messages=[
        {"role": "system", "content": "You are a helpful tutor."},
        {"role": "user",   "content": "Explain photosynthesis simply."}
    ],
    temperature=0.7,            # 0 = focused, 1+ = creative

    stream=False                # True = tokens arrive one-by-one (feels faster)
)

print(response.choices[0].message.content)

You pass an array of messages with roles (system / user / assistant) plus parameters that shape the reply — then read back the next response.

Two traps worth calling out live. First, tool calling never runs your function — the model only returns a JSON request; your code executes it and hands the result back. Second, "JSON mode" guarantees the reply is valid JSON, not that it matches your schema, so you still validate. Project 01's offline demo prints a MOCK banner and fakes these calls; only with a real key does it exercise an actual model.
02

The Customer Support Agent

From answering questions to taking actions
02

Agent vs. chatbot: the autonomy line

A chatbot generates text. An agent can act — look up an order, search docs, file a ticket — then decide what to do next on its own.

Analogy

A librarian who recommends books from memory (chatbot) vs. an assistant who walks the shelves, places holds, and renews loans (agent).

Aha

The moment a system can execute something instead of just describing it, it crossed from chatbot to agent.

02

The ReAct loop: reason → act → observe

The agent reasons about the next step, acts by calling a tool, observes the result — and repeats until done. Grounding each step in real results curbs hallucination.

Analogy

A detective: form a hypothesis, chase a lead, examine the evidence, revise. Not solved in one shot — solved by iterating.

Aha

The "thinking" happens between actions. No perfect upfront plan needed — it adapts as observations arrive. (Yao et al., ReAct 2022)

02

Tools are the agent's hands — and you write the loop

Tools (lookupOrder, searchFAQ, createTicket) are described like API docs; the model picks one by matching intent. A while loop runs tools until the model says "I'm done."

Analogy

Apps on a phone. The agent reads each app's description and opens the right one for the task at hand.

Aha

Decision quality lives in your tool descriptions. And you must cap iterations — or a confused agent loops forever.

02

Grounding & knowing when to escalate

Grounding = answer from retrieved real data, not from memory. Good agents also have guardrails: when out of depth, hand off to a human via a tool like createTicket.

Analogy

A self-driving car that hands control back in a snowstorm. Autonomy is great; knowing your limits is the real skill.

Aha

Escalation is designed in, not a failure. The agent recognizing its boundary and routing to a human is correct behavior.

02

The idea, in code

routing a request

def handle_customer_request(user_message):
    # 1. Figure out what the user actually wants
    intent = classify_intent(user_message)

    # 2. Route to the handler that matches the intent
    if intent == "order_status":
        response = check_order_status(user_message)
    elif intent == "faq":
        response = answer_faq(user_message)
    elif intent == "human_handoff":
        response = escalate_to_human(user_message)   # knows its limits
    else:
        response = "I'm not sure how to help with that."

    # 3. Hand the response back to the user
    return response

A real agent routes each request to the right action based on detected intent — then acts and responds, instead of just generating text.

Honest caveat if you run project 02 offline: its keyword mock planner returns "done" after the first pass, so mock mode only ever runs one iteration — it shows the loop's shape, not real multi-step reasoning. The genuine reason → act → observe loop only fires in live mode. This slide's routing snippet is deliberately simplified; the real payoff is the while loop with an iteration cap so a confused agent can't spin forever.
03

Exploring Embeddings

Turning meaning into geometry
03

An embedding is meaning as a list of numbers

Text becomes a vector (e.g. 384 numbers) positioned so that similar meanings sit close together. The model learns these positions from billions of examples — nobody hand-codes them.

Analogy

GPS coordinates for meaning. Just as lat/long places cities, embeddings place "dog" near "puppy" and far from "skyscraper".

Aha

Meaning is inferred from context — "a word is known by the company it keeps." Similar contexts → nearby vectors.

03

Directions carry meaning — you can do algebra

In vector space, consistent relationships become consistent directions. The classic: king − man + woman ≈ queen. "Gender" and "capital-of" are repeatable vector moves.

Analogy

A map where "north" always means more formal, "east" more positive. Moving northeast takes "okay" → "excellent".

Aha

You can literally do arithmetic on meaning: France − Paris + London ≈ England.

03

Cosine similarity: angle, not length

Cosine similarity measures the angle between two vectors (−1 to +1), ignoring their length. That's why it's the go-to for text.

Analogy

Two arrows pointing the same way are "similar" even if one's twice as long — like a 10-page essay and a 1-line summary on the same topic.

Aha

Euclidean distance would unfairly punish long documents. Cosine ignores size and focuses on topic direction.

03

Semantic search beats keyword search

Keyword search needs the exact words. Semantic search embeds the query and finds the nearest vectors — so "dog obedience classes" matches "puppy training" with zero shared words.

Analogy

A cookbook index that knows "crimson apple pie" answers "red fruit dessert" — no thesaurus required.

Aha

Synonyms, paraphrases, and intent are handled automatically. This is the engine under "related items" and modern search.

03

Local vs. API embeddings

A local model (downloaded once, ~25MB) runs offline, free, private, instant. An API model is often larger/sharper but needs network, costs per call, and sends your data out.

Analogy

Owning a calculator (local) vs. calling a math hotline (API). One works on a plane; the other always has the latest tricks.

Aha

The trade-off isn't only cost vs. quality — it's also privacy and latency, which often decide it.

03

The idea, in code

text → vectors → closeness

# embed() turns text into a vector. (Real ones are ~1536 dims; toy values here.)
vectors = {"dog":        [0.80, 0.90, 0.10],
           "puppy":      [0.75, 0.85, 0.15],
           "skyscraper": [0.10, 0.20, 0.90]}

def cosine_similarity(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# Similar meanings land near each other in vector space
print(cosine_similarity(vectors["dog"], vectors["puppy"]))       # ~0.99  close!
print(cosine_similarity(vectors["dog"], vectors["skyscraper"]))  # ~0.45  far

Text becomes numbers (an embedding), so "meaning" turns into distance — close vectors mean similar ideas, with zero shared keywords.

Project 03 runs a real local model (Xenova all-MiniLM-L6-v2, 384-dim) offline, so the semantics are genuine — the query "coding in a language named after a snake" ranks the Python line first with zero shared words. Two gotchas: the hash-fallback mode has no semantic meaning (identical strings score 1.0, everything else ~0), and vectors from different embedders (384-dim vs OpenAI's 1536-dim) can't be compared — cosineSimilarity throws on a dimension mismatch, so swapping models means re-embedding the corpus.
04

RAG over YouTube Transcripts

Giving an LLM an open book
04

Why RAG exists

LLMs have a knowledge cutoff, can't see your private data, and hallucinate when unsure. RAG (Retrieval-Augmented Generation) fixes this by retrieving real documents before answering.

Analogy

A closed-book exam (plain LLM) vs. an open-book exam (RAG). The student still thinks — but is anchored to real sources.

Aha

RAG doesn't make the model smarter — it makes it more honest: "I found this in document X." (Lewis et al., 2020)

04

The pipeline: chunk → embed → store → retrieve → generate

Documents are split, embedded, and indexed. At query time you embed the question, fetch the closest chunks, paste them into the prompt, and let the model answer from them.

Analogy

Running a library: catalog books (chunk + embed), shelve them (store), help patrons find sources (retrieve), write the paper (generate).

Aha

RAG is a pipeline, not one trick. A perfect embedding model still fails if your chunking is bad.

04

Chunking: size and overlap

Split text into focused 200–500 token pieces so retrieval can pinpoint the right paragraph. Add a small overlap so ideas spanning a boundary aren't cut in half.

Analogy

Indexing each recipe separately instead of searching one giant 500-recipe PDF for "caramelize onions".

Aha

Overlap matters as much as size — without it, a sentence stranded at a boundary never matches the query.

04

Grounding, citations & why we don't stuff everything

Tell the model to answer only from retrieved context and cite sources (e.g. [Video @ 01:23]). We retrieve a few best chunks instead of pasting the whole corpus — for cost and focus.

Analogy

A lawyer citing exhibits, not saying "trust me." And smart packing: bring the hammer you need, not the whole hardware store.

Aha

LLMs get "lost in the middle" of long contexts. Fewer, better chunks often beat a giant context window.

04

RAG vs. fine-tuning

RAG injects fresh knowledge at query time (easy to update). Fine-tuning bakes behavior/style into the weights. Start with RAG; fine-tune for how to answer, not what.

Analogy

RAG = a reference manual you consult on the job. Fine-tuning = sending someone to trade school.

Aha

They're complementary: fine-tune the voice, use RAG for the facts.

04

The idea, in code

the RAG pipeline

# 1. Index your documents (done once, ahead of time)
chunks = split_document(long_doc)     # break into bite-size pieces
index  = embed_and_store(chunks)      # vectors → searchable store

# 2. At query time, retrieve the most relevant chunks
question = "What is photosynthesis?"
top_chunks = retrieve(question, index, k=3)   # 3 closest pieces

# 3. Generate an answer GROUNDED in what we retrieved
prompt = f"""Answer using ONLY this context:
{top_chunks}

Question: {question}"""
answer = llm(prompt)                  # answers from docs, not memory

Retrieval-Augmented Generation fetches real chunks first, then asks the model to answer only from them — grounding it in sources instead of memory.

Key takeaway to stress: retrieval quality is the ceiling on answer quality — a perfect LLM can't fix bad chunks. Note for anyone running project 04: its demo hard-codes the 128-dim hash embedder and defaults to a MOCK generator that just stitches the first two sentences of each retrieved chunk plus a citation, so out of the box it demonstrates the pipeline shape with the weakest fallback at both ends. Real semantic RAG needs a one-line edit to drop the hash embedder and a live key.
05

Enterprise Advanced RAG

When "just embeddings" isn't enough
05

Why naive RAG breaks at scale

Pure semantic search fails when users say "time off" but the doc says "PTO", or when exact acronyms matter. Semantic similarity ≠ actual usefulness.

Analogy

A librarian who only understands themes but never checks the exact label on the shelf.

Aha

Two docs can mean similar things yet differ wildly in how useful they are for this question.

05

BM25: the keyword workhorse

BM25 scores documents by rare-word importance, term frequency (with diminishing returns), and length normalization. It's "bag of words" — order and meaning don't matter.

Analogy

Scoring reviews for "wood-fired pizza": rare terms count more, the 5th mention adds little, long reviews don't auto-win.

Aha

"PTO policy" and "policy for PTO" score identically — BM25 catches exact terms an embedding might gloss over.

05

Hybrid retrieval + Reciprocal Rank Fusion

Run keyword (BM25) and dense (embedding) search together. RRF merges their ranked lists using only positions (1/(k+rank)) — no fragile score-scaling needed.

Analogy

Two detectives — one reads faces (dense), one matches fingerprints (keyword). RRF is the panel that rewards their agreement.

Aha

RRF rewards consensus: docs both methods rank highly float to the top, even on totally different score scales.

05

Reranking: precise second pass

After retrieval returns a shortlist, a reranker rescores it for precision. In production that's a cross-encoder reading query and candidate together — far more accurate than the bi-encoder used to retrieve, but too slow to run on everything. This project keeps it simple: reranking.ts is a heuristic (query-term overlap + title match + recency bonus), a stand-in for the real learned cross-encoder.

Analogy

Bi-encoder = speed dating (2 min each, compare later). Cross-encoder = a full dinner date. Speed-date 100, dine with the top 10.

Aha

Two stages either way: fast retrieve → slow rerank. The heuristic teaches the shape of a rerank pass; a cross-encoder is the production upgrade that fills the same slot.

05

Query expansion & metadata filtering

Query expansion rewrites/enriches the question (add synonyms, or HyDE: generate a fake ideal answer and search for docs like it). Metadata filtering restricts by department, date, or type first.

Analogy

Netflix search with filters: "space adventure" + Sci-Fi + PG-13 + year≥2020 cuts the noise instantly.

Aha

HyDE works even if the fake answer hallucinates — the embedding still lands in the right neighborhood of real docs.

05

The idea, in code

fusing two searches

# Run BOTH searches — they're good at different things
keyword_results  = bm25_search(query)     # ["doc3", "doc1", "doc5"]
semantic_results = vector_search(query)    # ["doc1", "doc2", "doc3"]

# Reciprocal Rank Fusion: merge by POSITION, not raw scores
k = 60
scores = {}
for ranked_list in (keyword_results, semantic_results):
    for rank, doc in enumerate(ranked_list, start=1):
        scores[doc] = scores.get(doc, 0) + 1 / (k + rank)

# Docs that BOTH methods rank highly rise to the top
final = sorted(scores, key=scores.get, reverse=True)
# → ["doc1", "doc3", ...]   doc1 was near the top of both lists

Reciprocal Rank Fusion adds 1/(k+rank) from each list — rewarding consensus, with no fragile mixing of mismatched score scales.

The k=60 here matches the project's real value (hybrid-retrieval.ts, "typically 60"). Honest framing for the demo: on project 05's small, well-separated 10-doc corpus the naive retriever already ranks the right doc first, so the advanced stack shows no dramatic win — these techniques earn their keep on harder corpora. Also note the reranker's recency bonus only fires for docs under 365 days old, but every corpus doc is dated 2024 or earlier, so as of today that branch is dead code — a real lesson about wall-clock-tied scoring silently decaying as data ages.
06

A Simple Neural Network

How "learning" actually happens
06

A neuron is a weighted vote

Each neuron multiplies its inputs by learned weights, sums them, adds a bias, and passes the result through an activation. The weights are what training tunes.

Analogy

A committee member weighing evidence — some clues matter more — then voting yes/no once the total crosses a threshold.

Aha

A neuron fires on a weighted combination, not raw inputs. Change the weights, change the behavior.

06

Layers + non-linear activations = power

Stacked layers transform data step by step. Activation functions (ReLU, sigmoid) add non-linearity — without them, 10 layers collapse into 1 straight line.

Analogy

An assembly line: edges → shapes → objects. ReLU is a bouncer ("must be this tall"); sigmoid is a dimmer switch.

Aha

The non-linear "two-moons" dataset needs hidden layers — a straight line can never separate two interlocking crescents.

06

Loss: a number for "how wrong"

The loss function measures the gap between predictions and truth. Training = making this number as small as possible. Confident-and-wrong is punished hardest.

Analogy

Distance from the bullseye in darts. Outer ring = high loss; near the center = low loss.

Aha

Loss captures how wrong, not just if wrong — predicting 90% on the wrong answer hurts more than 51%.

06

Gradient descent + backprop: learning by going downhill

Gradient descent treats loss as a landscape and steps downhill. Backpropagation works backwards to assign each weight its share of the blame, so the right knobs get adjusted.

Analogy

A blindfolded hiker feeling the slope and shuffling downhill. Backprop = reviewing a relay race in reverse to see who slowed the team.

Aha

The network doesn't solve for the answer like algebra — it searches, step by step. That's why training takes many iterations.

06

Epochs, batches, learning rate

An epoch is one full pass over the data, processed in batches. The learning rate is step size — too small is slow, too big overshoots.

Analogy

Flashcards: an epoch is one run through the deck; batches are 10 cards at a time; learning rate is how hard you change tactics.

Aha

More epochs aren't always better — past a point the model overfits. Stop when loss converges.

06

The idea, in code

learning by gradient descent

# A neuron = weighted sum of inputs, squashed by an activation
def neuron(x, w, b):
    return 1 / (1 + np.exp(-(np.dot(w, x) + b)))   # sigmoid

w, b, lr = np.array([0.1, -0.2]), 0.0, 0.1         # start random
x, y_true = np.array([2.0, 3.0]), 1.0

# Training loop: predict → measure error → nudge weights downhill
for epoch in range(1000):
    y_pred = neuron(x, w, b)
    loss   = (y_pred - y_true) ** 2                # "how wrong"
    grad   = 2 * (y_pred - y_true) * y_pred * (1 - y_pred)
    w -= lr * grad * x                             # step toward less loss
    b -= lr * grad

Learning isn't solved like algebra — the network searches: predict, measure loss, nudge every weight a little downhill, repeat.

Live-demo warning worth internalizing: "seeded" is not the same as "reproducible." Project 06 seeds both the dataset and the weight initialization, yet tfjs's per-epoch mini-batch shuffle is a separate unseeded randomness source, so final loss still lands in two clusters (~0.05 at 100% accuracy, or ~0.22 at ~90%) across runs. That's why the project quotes ranges rather than one hard-coded number — any pasted exact loss curve that never reproduces is a slop tell.
07

A Claude-Code-like Coding Agent

An agent with hands — safely
07

A coding agent is a planner with a feedback loop

Give an LLM a folder, tools (read/write file, run command), and a goal in plain English. It plans, edits, runs, reads results, and iterates until done.

Analogy

An intern who only touches your codebase through a keyboard: explores, writes, tests, fixes, reports back.

Aha

It's not "autocomplete on steroids." The power is observing what breaks and adjusting — not generating code blindly.

07

Tools are hands; feedback is sight

The agent lives in text (tokens in/out); tools execute in the real world and report back. Running tests and reading errors is how it grounds itself in this codebase.

Analogy

A remote surgeon driving a robot arm by command and watching the feed — and IKEA furniture: you only learn the holes don't line up by trying.

Aha

Agents get smarter by trying things and reading failures, not by reading more docs.

07

Sandboxing & path jailing

Constrain the agent to one workspace folder. Beware path traversal: ../../etc/passwd escapes the jail. The fix: resolve the path to absolute, then verify it's still inside the sandbox.

Analogy

A child in a fenced backyard sandbox — free to build, but the fence keeps them out of traffic.

Aha

Banning the literal ".." isn't enough (attackers URL-encode it). Trust the resolved destination, not the input string. (OWASP)

07

Human-in-the-loop & context management

Risky actions (delete, push) pause for human approval. And since context windows are finite, long tasks need pruning: summarize done work, offload progress to files.

Analogy

A self-driving car that asks you to take the wheel in tricky spots. And taking notes instead of holding everything in your head.

Aha

The best agents don't fight the context limit — they "think on paper," using files as external memory.

07

The idea, in code

the agent loop + sandbox

while not task_done:
    # OBSERVE: read files, last test run, error messages
    context = {"files": read_files(), "last_result": observe()}
    action  = model.decide(context)        # pick: read / edit / run_tests

    if action.type == "edit":
        if not is_safe_path(action.path, workspace="/project"):
            raise SecurityError("Path escapes the workspace!")
        write_file(action.path, action.content)
    elif action.type == "run_tests":
        context["last_result"] = run_sandboxed(action.command)  # feed back

def is_safe_path(path, workspace):
    root     = os.path.realpath(workspace)
    resolved = os.path.realpath(path)      # trust the RESOLVED path, not the raw string
    # Bare prefix isn't enough: "/project-evil".startswith("/project") is True.
    return resolved == root or resolved.startswith(root + os.sep)  # sibling can't escape

The agent grounds itself by reading real failures and looping — while a sandbox check on the resolved path keeps it from escaping the workspace.

This is_safe_path is the corrected version. The real project 07 originally shipped two exploitable bugs in its advertised "safety features": a startsWith path jail with no trailing separator let a sibling dir like /tmp/sandbox-evil escape a /tmp/sandbox jail, and a first-token command whitelist was bypassed because the full string was later handed to a shell (node x.js; whoami). Both are fixed now with regression tests. The deeper lesson: even after fixing those, node -e runs arbitrary JS, so a string whitelist is a speed bump, not OS-level isolation.
08

The AI Code Reviewer

Rules + judgment, working together
08

The diff: a universal language for change

A unified diff shows what changed: + added, − removed, plain = context. @@ marks a hunk with line numbers, so any tool applies the change to the right spot.

Analogy

Track-changes in a doc — but encoded as machine instructions, not visual strikethrough.

Aha

A hunk includes a few unchanged context lines on purpose — so the patch still lands even if nearby lines shifted.

08

Deterministic linting vs. LLM judgment

A linter applies fixed rules — same input, same output, instant, free. An LLM reasons about intent: unclear names, missing edge cases, risky logic.

Analogy

Spellcheck (linter) catches "teh" every time. An editor (LLM) tells you the argument doesn't make sense.

Aha

Detecting all bugs statically is provably impossible (Rice's theorem) — so deterministic tools trade completeness for reliability.

08

Why combine them + structured output

Layer them like a test pyramid: cheap linter rules catch clear violations; the LLM catches subtle ones. Force the LLM to return a schema (file, line, severity, suggestion) so results feed dashboards and CI.

Analogy

Car safety: airbags + lane-departure warnings. Each catches what the other can't. Structured output = a filled-in form, not prose.

Aha

If the LLM flags something a regex could have caught — write the regex. Push checks as low (fast/cheap) as possible.

08

Limits: false positives & humans in the loop

Both linters and LLMs produce false positives (and miss real bugs). There's no setting that minimizes both — it's a sensitivity/specificity trade-off. Humans dismiss noise and make judgment calls.

Analogy

Airport metal detectors: too sensitive flags every belt buckle; too lax misses real threats. Layers + a human screener handle it.

Aha

Too many false positives cause alert fatigue — people ignore all warnings, including the real ones. Trust is the metric.

08

The idea, in code

lint + LLM, structured

# Layer 1 — deterministic linter: fast, fixed rules, same result every time
def lint(code):
    findings = []
    if re.search(r"console\.log", code):
        findings.append({"line": 5, "severity": "warning",
                         "message": "Remove console.log"})
    return findings

# Layer 2 — LLM: judges intent, naming, risky logic a regex can't see.
# Forced to return the SAME schema {line, severity, message}.
def llm_review(code):
    return ask_llm(f"Review this code. Reply as JSON findings:\n{code}")

# Merge both into one structured list → feeds CI & dashboards
findings = lint(code) + llm_review(code)

Cheap rules catch the obvious; the LLM catches the subtle. Forcing both into one schema (not prose) lets the results gate CI together.

Point out the two opposite failure modes honestly. Project 08's regex linter has no lexer, so it false-positives on == inside strings/comments and flags JS-only patterns on valid Python — the inherent ceiling of regex vs AST linting. And its offline "LLM review" is filename-pattern-matched canned text keyed to the two sample diffs (auth.ts / user-handler.js); any other diff yields zero Source:llm comments, so the mock shows the pipeline, not analysis. The source tag on each comment is the honest signal of which trust it deserves.
09

The Voice AI Agent

Ears → brain → mouth
09

Three stages: STT → LLM → TTS

Voice agents split a hard task into three swappable parts: Speech-to-Text (ears), the LLM (brain), and Text-to-Speech (mouth). Each does one job and snaps together.

Analogy

A kitchen with three stations — take orders, cook, plate. Back one up and you add staff there without touching the others.

Aha

Modularity isn't future-proofing — it makes the whole pipeline testable offline with mock stages, no API keys.

09

Digital audio: a WAV is just numbers

Sound is a continuous wave; computers sample its height thousands of times per second (44,100 Hz for CD quality) and store the raw numbers (PCM). A WAV = a ~44-byte header + those samples.

Analogy

A flipbook: many still frames create smooth motion. More samples per second = smoother sound.

Aha

Nyquist: sample at ≥2× the highest frequency you want. Human hearing tops ~20kHz → 44.1kHz standard.

09

Latency & turn-taking: why voice is hard

Humans expect a reply within ~1 second. Three stages stacked easily hit 3–5s. VAD (voice activity detection) must also decide when you've actually stopped talking vs. just paused.

Analogy

A phone call with a 5-second gap before every reply feels like the line dropped. Silence reads as failure.

Aha

The fix is streaming/overlap, not faster servers — start each stage before the previous fully finishes.

The idea, in code

STT → LLM → TTS

class VoiceAgent:
    def __init__(self, stt, llm, tts):
        self.stt = stt   # ears   — swap Whisper, a mock, ...
        self.llm = llm   # brain  — swap Claude, GPT, local ...
        self.tts = tts   # mouth  — swap ElevenLabs, a mock ...

    def respond(self, audio_in):
        text      = self.stt.transcribe(audio_in)   # audio → text
        reply     = self.llm.respond(text)          # text  → text
        audio_out = self.tts.speak(reply)           # text  → audio
        return audio_out
        # latency stacks across stages → stream/overlap in production

agent = VoiceAgent(stt=Whisper(), llm=Claude(), tts=ElevenLabs())

Voice = three swappable stages: ears → brain → mouth. Each is just an interface, so you can mock one to test offline with no API keys.

The swappable-interface seam is the real lesson: an implementation-blind VoiceAgent chains three one-method interfaces. In project 09's mock, STT is just readFile, the LLM is five keyword rules, and TTS is a 440Hz sine tone — so the mock verifies the wiring and live verifies the intelligence. Worth flagging that a file's extension is a contract about its bytes: OpenAI TTS defaults to MP3, so naming the output .wav without setting the wav format produces a subtly broken file — a bug that was fixed in this project.

The AI Trading Agent

Signals, strategies & the limits of backtests
⚠️ Educational simulation only. Synthetic data, no live markets, not financial advice. The lesson here is as much about pitfalls as about indicators.
10

Indicators: SMA, EMA, RSI

Indicators turn price history into signals. SMA averages the last N prices (smooths noise). EMA weights recent prices more (reacts faster). RSI (0–100) flags overbought (>70) / oversold (<30) momentum.

Analogy

A car dashboard. SMA = your weekly step average; EMA = trusting today's forecast over last week's; RSI = a tachometer's RPM.

Aha

Every moving average lags — it's backward-looking by design. Smoothness and responsiveness are a trade-off.

10

Strategies are reactive, not predictive

A strategy is fixed rules turning signals into buy/sell/hold (e.g. a "golden cross" when a fast average crosses above a slow one). It removes emotion and guesswork.

Analogy

A recipe — consistent steps, no guarantee of a great meal. Two runners: the signal fires when the fast one overtakes the slow one.

Aha

Crossovers are lagging: the trend already started when the signal appears. You buy confirmation, not a head start.

10

Backtesting & its traps

Backtesting simulates a strategy on historical data. Beware overfitting (fitting noise), look-ahead bias (using future data), and survivorship bias (only testing winners).

Analogy

Memorizing last year's exam answers — perfect retest, fail on new questions. Or interviewing only lottery winners about "strategy."

Aha

Backtesting is "predicting the past." A flawless backtest is often answering the wrong question — past performance ≠ future returns.

10

The idea, in code

the crossover signal

def sma(prices, n):
    return sum(prices[-n:]) / n          # average of the last n prices

def signal(prices):
    # Compare a FAST (10-day) vs SLOW (30-day) average, now vs the prior bar
    fast,      slow      = sma(prices, 10),     sma(prices, 30)
    prev_fast, prev_slow = sma(prices[:-1], 10), sma(prices[:-1], 30)

    if prev_fast <= prev_slow and fast > slow:
        return "BUY"     # golden cross: fast just crossed ABOVE slow
    elif prev_fast >= prev_slow and fast < slow:
        return "SELL"    # death cross:  fast just crossed BELOW slow
    return "HOLD"        # no crossover — moving averages LAG the trend

A strategy is just fixed reactive rules on smoothed prices. The crossover confirms a trend already underway — it doesn't predict. (Educational only.)

These 10/30 periods match what project 10's demo actually runs (FAST_PERIOD=10, SLOW_PERIOD=30) — the original code once fed the LLM prompt mislabeled 20/50 periods, a bug now fixed. Foreground the headline result honestly: the strategy underperforms buy-and-hold (+20.95% vs +42.91%), and that's the intended lesson, not a defect — a trend-follower sits in cash during the early flat stretch and loses to passive holding in a steady uptrend. The whole point of separation: the LLM commentary is a passenger that never touches the deterministic trading path.

Multi-Input OCR

Fusing two kinds of information
11

OCR & why images are special

OCR turns images of text into editable text. Images carry spatial meaning — a pixel only matters relative to its neighbors — so you can't shuffle them like spreadsheet columns.

Analogy

A book with every letter randomly scattered: all the ink is there, but the structure that makes words is gone.

Aha

Flattening an image into a flat list throws away "who's next to whom" — the very thing that makes it readable.

11

CNNs: sliding pattern detectors

A CNN slides small filters (tiny "magnifying glasses") across the image, reusing the same filter everywhere. Early layers find edges; deeper layers combine them into shapes and objects.

Analogy

A detective with an "edge" stencil sliding over a photo, marking every match — while colleagues use other stencils.

Aha

Parameter sharing (same filter everywhere) is why CNNs learn efficiently — a useful edge detector helps at every location.

11

Two branches, one head: feature fusion

A multi-input model runs each input through its own branch, then concatenates them into a shared head for the final decision. Real OCR would put a CNN in the image branch; this project flattens the tiny 14×14 digit into a dense branch (model.ts: Input → Flatten → Dense(64), no Conv2D) to keep the focus on the fusion, not the CNN.

Analogy

Picking a restaurant: one friend reads reviews (text), another studies food photos (images). You merge both to decide.

Aha

You let each input speak its own "language" before merging — not just stapling raw pixels and numbers together upfront.

11

Why functional, not sequential

A Sequential model is one straight stack — single input, single output. Branching needs the Functional API, which treats the model as a graph with multiple input nodes.

Analogy

A single-file trail (Sequential) vs. a trail network that splits and merges (Functional).

Aha

Multi-input isn't exotic — it's everywhere: medical AI (scan + history + labs), self-driving (camera + LiDAR + GPS).

11

The idea, in code

two branches, one head

# Branch 1 — the IMAGE speaks "pixels" (a CNN reads it)
image_input  = Input(shape=(64, 64, 3))
x            = Conv2D(32, (3, 3), activation="relu")(image_input)
image_branch = Dense(64, activation="relu")(Flatten()(x))

# Branch 2 — the EXTRA features speak "numbers"
aux_input  = Input(shape=(5,))
aux_branch = Dense(32, activation="relu")(aux_input)

# Let each branch learn first, THEN fuse the representations
merged = concatenate([image_branch, aux_branch])
output = Dense(10, activation="softmax")(merged)

# Functional API: a model with TWO inputs — Sequential can't do this
model = Model(inputs=[image_input, aux_input], outputs=output)

Two branches, one head. Each input is processed in its own "language" before being concatenated — which needs the functional API, not a single stack.

Two honesty notes. First, the Conv2D line above is the general multi-input pattern — the actual project skips it: model.ts is Input(14×14×1) → Flatten → Dense(64) with no convolution, because on a tiny 14×14 digit the CNN waste is small and the teaching budget is spent on the fusion step. Second, be honest about the demo's headline 100% accuracy: a per-branch ablation shows image-only also hits 100% and aux-only ~99%, so each input already solves this toy task alone and the second branch buys nothing here. That's the actual teachable point — a headline accuracy is meaningless without baselines. On real data (medical scan + labs, camera + LiDAR) the fusion genuinely matters; this toy dataset just makes the pattern visible.

The MNIST Autoencoder

Compress to understand
12

MNIST & the idea of an autoencoder

Real MNIST = 70,000 handwritten digits (0–9), the "hello world" of vision. To stay fully offline this project fakes it: data.ts draws 400 synthetic 28×28 digits of just three classes (0, 1, 2). An autoencoder then learns to compress its input and reconstruct it — its job is simply to copy itself through a narrow squeeze.

Analogy

Summarize a 10-page story into one paragraph, then rewrite the full story from just that summary.

Aha

It's not really "unsupervised" — it's self-supervised: the input is its own label. And a synthetic stand-in is a fair swap here — the squeeze lesson is identical whether the digits are real or drawn.

Honesty note on the data: to run fully offline this project does not download real MNIST. Its data.ts draws 400 synthetic 28×28 digits of only three classes (0, 1, 2) with seeded randomness — a same-shaped stand-in, not the real 70,000-image 0–9 set. Say "MNIST-like" when demoing it. The autoencoder lesson (the bottleneck squeeze) is identical either way, which is why the swap is fair rather than a shortcut worth hiding.
12

Encoder → bottleneck → decoder

The encoder shrinks the image (784 → 32), the bottleneck holds the compressed code, and the decoder expands it back. The squeeze forces the network to keep only what matters.

Analogy

Packing for a trip with a tiny suitcase — you bring versatile essentials (edges, curves), not every detail.

Aha

Too-wide a bottleneck lets it cheat by memorizing; too narrow and it can't reconstruct. The squeeze is the lesson.

12

Latent space & reconstruction loss

The latent space is those bottleneck numbers — abstract features, not pixels. Reconstruction loss (output vs. input) is the training signal pulling encoder and decoder to improve together.

Analogy

A recipe captures how to recreate a dish, not every molecule. Similar digits cluster nearby in latent space.

Aha

Encoder and decoder are partners, not competitors — one loss number improves both at once.

12

vs. PCA, and what it's good for

Unlike linear PCA, autoencoders learn non-linear compression — better for curvy data like images. Real uses: denoising, anomaly detection, and data-specific compression.

Analogy

PCA folds a map along straight creases; an autoencoder crumples it smartly to follow the terrain's curves.

Aha

They're specialists: a digit autoencoder fails on cat photos — and that failure is exactly what powers anomaly detection.

The idea, in code

compress, then reconstruct

# ENCODER: squeeze 784 pixels → 32 numbers (the bottleneck)
input_img = Input(shape=(784,))
encoded   = Dense(128, activation="relu")(input_img)
encoded   = Dense(32,  activation="relu")(encoded)   # keep only what matters

# DECODER: expand 32 numbers back → 784 pixels
decoded = Dense(128, activation="relu")(encoded)
decoded = Dense(784, activation="sigmoid")(decoded)

autoencoder = Model(input_img, decoded)
autoencoder.compile(optimizer="adam", loss="mse")
autoencoder.fit(X_train, X_train, epochs=50)   # the target IS the input!

784 → 32 → 784. Trained to copy its input through a narrow squeeze — too wide and it memorizes, too narrow and it can't rebuild. Input is its own label (self-supervised).

A subtle wiring bug worth mentioning: project 12 originally displayed the latent vector from a standalone untrained encoder while training a separate autoencoder with independent weights — so its "what the model learned" visualization was actually random noise. The fix shares the encoder/decoder layer instances at construction and keeps a handle on the sub-model. Lesson: green tests and a running demo can still hide a wiring error — the demo was measuring a different network than the one it trained. Loss is given as a range because weight init here is unseeded.
